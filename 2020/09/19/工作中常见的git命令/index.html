最近由于工作的原因对git命令有了新的认识，今天想全部总结一下，也算是做个笔记，后期有新的见解又继续添加。
终于又更新了一部分了，希望这些对自己有用。

基本认识

拉取项目

    git clone [url,ssh]

git init 初始化一个仓库

$ mkdir git-tutorial
$ cd git-tutorial
$ git init
Initialized empty Git repository in /Users/hirocaster/github/github-book
/git-tutorial/.git/

git add 向暂存区添加文件

    git add README.md

提交暂存区到本地仓库区

    git commit -m [message]
    // 如果需要详细的描述可不加 -m ，之后弹出编辑框编辑后保存提交

git log 查看提交的日志信息

    git log
    // 想要只显示提交的描述，在log后加上 --pretty=short
    // 想要查看提交的文件改动为： git log -p 默认为查看所有改动
    // 查看指定文件改动日志：git log -p README.md

git diff 查看更改前后的差别

    git diff 默认查看当前树和(暂存区,最新提交状态)的差别
    git diff HEAD 查看当前提交于上次提交的差别

查看修改文件状态

    git status

查看所有分支

    git branch -a

创建一个新分支并切换到改该分支

    git checkout -b [branch name]

切换分支

    git checkout [branch name]

拉取远程仓库代码并与本地分支合并

    git pull [remote] [branch]

上传本地指定分支到远程仓库

    git push [remote] [branch]

删除本地分支

    git branch -d [branch name]

删除远程仓库分支(分支名前的冒号表示删除)

    git push origin :branchName

查看本地已添加的远程仓库

    git remote show

查看指定的本地关联的远程仓库remote地址，远程分支，还有本地分支与之相对应关系

    git remote show [origin]

与指定远程仓库分支比较，删除本地多余的分支

    git remote prune [origin]

进阶魔法

git marge 合并分支

    // 我们首先切换到master分支，
    git checkout master
    // 然后我们在将feature/A 分支合并到master分支
    git marge --no-ff feature/A
    // 注意这里为了明确本次合并记录需要加上--no-ff 随后会出现编辑器编辑完成保存提交就完成啦

git reset 回溯历史版本

之前我们学习的都是完成功能后提交代码，git的另一特性就现的更灵活。那就是可回溯代码。
示例：在master之前合并了feature/A分支，我们想要回到合并之前的代码。
要完全恢复到之前的代码，我们需要拿到目标时间点的哈希值(即提交的commitId)

    git reset --hard fd0cbf0d4a25f747230694d95cac1be72d33441d

git reflog 查看当前仓库执行过的操作的日志

在日志中，我们可以看到 commit、checkout、reset、merge 等 Git 命
令的执行记录。只要不进行 Git 的 GC（Garbage Collection，垃圾回收），
就可以通过日志随意调取近期的历史状态，就像给时间机器指定一个时
间点，在过去未来中自由穿梭一般。

    git reflog

遇到冲突如何解决

在我们执行git marge –no-ff feature/A 后，出现了冲突，
解决办法：我们打开编辑器，查看有冲突的地方，编辑后再次提交到feature/A分支，再次执行合并请求
冲突就解决了啦~。

如果我们对上一次的提交描述不满意怎么办

使用 git commit –amend
执行上面的命令后会出现编辑器，修改我们之前的描述，保存提交就可以啦~
使用 git log –graph 查看提交，修改就成功啦

git rebase -i 压缩历史提交

如果我们提交了两次代码到本地仓库，然后发现第一次提交的描述有错误的地方，我们可以这样做。
使用git rebase -i HEAD~2 命令
执行后会出现编辑器，显示如下

pick 7a34294 第二次提交描述
pick 6fba227 第一次提交描述
# Rebase 2e7db6f..6fba227 onto 2e7db6f
#
# Commands:
# p, pick = use commit
# r, reword = use commit, but edit the commit message
# e, edit = use commit, but stop for amending
# s, squash = use commit, but meld into previous commit
# f, fixup = like "squash", but discard this commit's log message
# x, exec = run command (the rest of the line) using shell
#
# These lines can be re-ordered; they are executed from top to bottom.
#
# If you remove a line here THAT COMMIT WILL BE LOST.
#
# However, if you remove everything, the rebase will be aborted.
#
# Note that empty commits are commented out

我直接讲第一次描述 6fba227 压缩到第二次提交描述(7a34294)中，修改第一次描述提交(6fba227)前的（pick）为（fixup）.
保存并关闭编辑器，系统提示rebase成功，那压缩就完成啦~
我们在使用 git log –graph 查看，发现我们第二次提交的commitId已经发生了改变，而第一次提交则别完全抹除掉了。
